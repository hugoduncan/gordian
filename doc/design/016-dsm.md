# 016 — Dependency Structure Matrix (DSM)

## Problem

Gordian already exposes architectural structure through graphs, metrics,
findings, subgraphs, and communities. But some structural questions are easier
to answer with a matrix than with a graph:

- what are the strongly coupled knots in the system?
- what does the architecture look like after cycles are collapsed?
- is the overall shape mostly layered or broadly cross-cutting?
- where are the major inter-component dependencies once internal tangles are
  factored out?

A raw namespace-by-namespace matrix is theoretically complete, but in practice
it becomes unreadable quickly. Most cells are empty, and for medium-to-large
projects the full matrix is too large for text or markdown output.

The useful architectural unit is not usually the individual namespace cell. It
is:
- the strongly connected block
- the relationships between those blocks
- the internal structure of the non-trivial blocks

## Goal

Add a new command:

```bash
gordian dsm
```

that presents architecture as:

1. a **collapsed SCC matrix** over the condensation graph
2. **detail mini-matrices** for each non-singleton SCC

This yields a scalable, explainable matrix view:
- the collapsed matrix gives global structure
- the SCC detail matrices give local cycle structure

---

## Scope

### In scope
- new `dsm` subcommand
- SCC decomposition of the project require graph
- collapsed SCC matrix over all SCCs, including singleton SCCs
- topological ordering of the collapsed SCC graph
- summary metrics for the collapsed matrix
- detail sections for non-singleton SCCs
- text, markdown, EDN, and JSON output
- machine-readable schema for collapsed matrix and SCC detail matrices

### Out of scope
- full namespace-level DSM as the default view
- residual DSMs (remove blocks entirely)
- community-based DSM blocking
- weighted conceptual/change DSMs
- automatic matrix reordering beyond SCC condensation + stable topo order
- CSV export
- gate/compare integration in v1

---

## Design choice: collapse, not full matrix

A full namespace-level DSM preserves every cell but scales poorly.

A residual DSM removes strong blocks and shows only what remains, but it loses
architecture relationships involving the removed blocks.

A collapsed DSM is the right abstraction for Gordian:
- dimensions shrink
- inter-component structure is preserved
- cycles are localized inside SCCs
- the collapsed graph is a DAG
- the output remains readable in text and markdown

This feature therefore treats SCCs as first-class architecture units.

---

## Conceptual model

Given directed require graph `G`:

1. compute strongly connected components `SCC(G)`
2. build the condensation graph `Cond(G)`
   - each SCC becomes one node
   - edges between SCCs are induced by original edges crossing SCC boundaries
3. topologically order `Cond(G)`
4. render:
   - the collapsed SCC matrix for `Cond(G)`
   - one internal matrix for each non-singleton SCC

This yields two complementary views:

### Collapsed SCC matrix
Answers:
- how do the major structural units depend on one another?
- is the architecture broadly feedforward after cycles are collapsed?
- which SCC blocks are large, dense, or central?

### SCC detail matrices
Answers:
- what does each cyclic knot actually look like internally?
- is the SCC a simple 2-node mutual dependency or a dense tangle?
- which members participate in the internal feedback?

## CLI

### New command

```bash
gordian dsm [dirs...] [options]
```

Examples:

```bash
gordian dsm
gordian dsm .
gordian dsm src/
gordian dsm . --markdown
gordian dsm . --json
gordian dsm . --include-tests
```

### Flags

```text
--json               JSON output
--edn                EDN output
--markdown           Markdown output
--include-tests      Include discovered test dirs
--exclude <regex>    Exclude namespaces (repeatable)
```

### Defaults
- use existing project auto-discovery rules
- analyze the structural require graph only
- compute SCCs over the filtered project graph
- render collapsed SCC matrix plus non-singleton SCC details

### Non-goals for v1 CLI
No new lens or threshold flags in v1. DSM is structural only in the first
implementation.

---

## Why not display the full matrix?

The full matrix is not the right default because:
- dimensions grow with namespace count
- readability collapses quickly beyond small projects
- most cells are empty
- the most important structure is block-level, not cell-level

Even hiding diagonal blocks while keeping the same namespace basis does not
solve the scale problem; dimensions remain unchanged.

The collapsed SCC matrix actually reduces dimensionality by changing the basis
from namespaces to SCC blocks.

---

## Matrix semantics

### Namespace graph
Input graph shape:

```clojure
{ns #{dep ...}}
```

### SCC block
An SCC block is a set of namespaces with mutual reachability.

Each namespace belongs to exactly one SCC.

Singleton SCCs are still SCC blocks and must be included in the collapsed
matrix.

### Collapsed matrix basis
Rows and columns are SCC blocks, ordered topologically over the condensation
DAG.

Cell `(Bi, Bj)` means:
- there exists at least one original edge from a namespace in `Bi`
  to a namespace in `Bj`
- `Bi != Bj`

### Cell value
For v1, a collapsed cell should carry at least:

```text
edge-count = number of original namespace-level edges crossing from Bi to Bj
```

This is more informative than a boolean cell while remaining simple.

---

## Ordering

### Recommendation
Order SCC blocks by:
1. topological order of the condensation DAG
2. stable deterministic tie-break among available zero-in-degree blocks

Tie-break recommendation:
- smallest lexicographic member name within the SCC

### Why
This gives:
- deterministic output
- a feedforward reading order
- stable snapshots for machine-readable output
- simple explanation to users

### Consequence
Because the SCC condensation graph is acyclic, the collapsed matrix can be
arranged in a feedforward order.

The interesting cycle structure therefore lives inside the SCC detail matrices,
not between collapsed blocks.

## Output model

The command should expose two first-class artifacts:

1. `:collapsed` — the collapsed SCC matrix
2. `:scc-details` — detail records for non-singleton SCCs

### Canonical machine-readable shape

```clojure
{:gordian/command :dsm

    :internal-edge-count 0

```

## Block metadata

Each block in `:collapsed :blocks` should include enough metadata to make the
collapsed matrix interpretable without requiring drilldown.

### Block fields

```clojure
{:id                  int
 :members             [sym ...]
 :size                int
 :cyclic?             boolean
 :internal-edge-count int
 :density             float}
```

### Semantics
- `:id` is the row/column identifier within the collapsed matrix
- `:members` are the namespaces in the SCC
- `:size` is the number of members
- `:cyclic?` is true when `:size > 1`
- `:internal-edge-count` counts namespace-level edges whose endpoints are both
  inside the SCC
- `:density` is the directed density of the internal SCC subgraph

### Density formula
For block size `n` and internal edge count `m`:

```text
density = m / (n * (n - 1))   when n > 1
density = 0.0                 when n = 1
```

No self-edges are counted.

---

## Collapsed edges

Collapsed edges summarize namespace-level dependencies between SCCs.

### Edge fields

```clojure
{:from       int
 :to         int
 :edge-count int}
```

### Semantics
For collapsed blocks `Bi` and `Bj`, `edge-count` is the number of original
namespace-level structural edges:

```text
u -> v such that u ∈ Bi, v ∈ Bj, and i != j
```

### Why count matters
A boolean cell only says that some relationship exists.
An edge count also says whether the block relation is:
- light and incidental
- concentrated
- broad and structurally significant

---

## SCC detail matrices

For each SCC with `size > 1`, include a detail record describing the induced
subgraph inside the SCC.

These are the mini-matrices that reveal the internal cycle structure hidden by
collapse.

### Detail fields

```clojure
{:id                  int
 :members             [sym ...]
 :size                int
 :internal-edges      [[i j] ...]
 :internal-edge-count int
 :density             float}
```

### Coordinate semantics
`[i j]` refers to the local member ordering within `:members`.

### Local ordering
Use deterministic member ordering:
- lexicographic namespace order

This keeps mini-matrices stable and easy to test.

---

## Metrics

### Collapsed summary metrics
Recommended v1 summary metrics:

```clojure
{:block-count          int
 :singleton-block-count int
 :cyclic-block-count   int
 :largest-block-size   int
 :inter-block-edge-count int
 :density              float}
```

### Metric meanings
- `:block-count` — number of SCCs in the filtered project graph
- `:singleton-block-count` — number of SCCs of size 1
- `:cyclic-block-count` — number of SCCs of size > 1
- `:largest-block-size` — maximum SCC size
- `:inter-block-edge-count` — number of structural edges crossing SCC boundaries
- `:density` — density of the collapsed DAG matrix

### Collapsed density formula
For block count `n` and inter-block edge count `m`:

```text
density = m / (n * (n - 1))   when n > 1
density = 0.0                 when n <= 1
```

Even though the collapsed graph is acyclic, directed density remains a useful
coarse measure of cross-block coupling.

## Text output

Text output should emphasize summary first, then readable detail.

### Proposed sections

1. **SCC decomposition summary**
2. **Collapsed SCC matrix**
3. **Cyclic SCC details**

### Example sketch

```text
SCC decomposition
  namespaces: 48
  blocks: 41
  cyclic blocks: 3
  largest block: 5

Collapsed SCC DSM
  inter-block edges: 57
  density: 0.0348

Blocks
  B0  size 1  cyclic no   members: gordian.aggregate
  B1  size 3  cyclic yes  density 0.83  members: foo.a, foo.b, foo.c
  B2  size 1  cyclic no   members: gordian.close

Block matrix
      B0 B1 B2 B3
B0     .  2  .  .
B1     .  .  1  .
B2     .  .  .  3
B3     .  .  .  .

Cyclic SCC B1
  size: 3
  members:
    foo.a
    foo.b
    foo.c
  internal edges: 5
  density: 0.83

  mini-matrix
      0 1 2
  0   . X X
  1   . . X
  2   X X .
```

### Rendering guidance
- if collapsed block count is small enough, render a literal matrix
- if too large, render a sparse inter-block edge list instead
- always render block metadata table/list
- always render non-singleton SCC detail summaries

The precise size threshold is an implementation detail and does not need to be
part of the public schema.

## Markdown output

Markdown should follow the same information architecture:

1. summary table
2. blocks table
3. collapsed block relation table or small matrix
4. SCC detail sections for non-singletons

### Suggested summary table

| Metric | Value |
|---|---:|
| Namespaces | 48 |
| SCC blocks | 41 |
| Cyclic SCCs | 3 |
| Largest SCC | 5 |
| Inter-block edges | 57 |
| Density | 0.0348 |

### Suggested block table

| Block | Size | Cyclic | Density | Members |
|---|---:|---|---:|---|
| B0 | 1 | no | 0.00 | `gordian.aggregate` |
| B1 | 3 | yes | 0.83 | `foo.a`, `foo.b`, `foo.c` |

### Suggested inter-block table

| From | To | Edge count |
|---|---|---:|
| B0 | B1 | 2 |
| B1 | B2 | 1 |

Markdown tables scale better than literal matrix rendering for larger block
counts.

## JSON / EDN schema notes

No schema bump is required merely to add a new command.

The envelope should identify:

```clojure
{:gordian/command :dsm}
```

The `dsm` command output should be self-contained and not depend on diagnose or
analyze-specific sections.

### Design principles
- make blocks first-class records
- keep edge coordinates explicit and stable
- keep non-singleton SCC details separate from the collapsed matrix
- ensure deterministic ordering for all sequences

## Relationship to existing Gordian features

### SCC detection
DSM reuses existing SCC analysis as its structural foundation.

### Explain
A future enhancement may allow:
- `gordian explain --block B7`
- or block-aware explain fallbacks

But that is not required for v1.

### Subgraph
A future enhancement may allow:
- `gordian subgraph <prefix> --dsm`
- or `gordian dsm <prefix>` over a subgraph basis

Not required for v1.

### Communities
A future version may introduce community-based blocking. That would be a
separate mode from SCC blocking, not a replacement for it.

### Compare / gate
Future integration could compare:
- block counts
- cyclic block counts
- largest SCC size
- collapsed density

But this design does not require compare/gate work in the first slice.

---

## Non-goals and rejected alternatives

### Full namespace matrix as default
Rejected because it is too large and low-signal for medium/large projects.

### Hide diagonal blocks but keep full namespace basis
Rejected as a default because it reduces clutter but not dimensions.

### Residual DSM
Rejected for v1 because removing strong blocks also removes their architectural
relationships.

### Community-first DSM
Rejected for v1 because SCCs are deterministic, already available, and give a
clean DAG-level collapse with no threshold tuning.

---

## Future extensions

### Full namespace DSM
Useful as an explicit low-level/debug/export mode.

### Community-blocked DSM
Alternative basis using detected architecture communities.

### Weighted DSMs
Allow cells to encode conceptual/change/combined strength.

### Block drilldown
Interactive or explicit drilldown for one SCC block or one collapsed edge.

### Pair examples on collapsed edges
Attach representative namespace-level edges for each block relation.

### Compare / gate support
Diff and ratchet checks over SCC/DSM metrics.

## Success criteria

The first DSM implementation is successful if a user can:

1. run `gordian dsm`
2. immediately see how many SCC blocks exist and how many are cyclic
3. understand the coarse architecture from the collapsed SCC matrix
4. inspect the internal structure of each non-trivial SCC
5. consume the same information through text, markdown, EDN, and JSON

The feature does not need a full namespace-level matrix to be useful.
Its value comes from choosing the right structural basis.
